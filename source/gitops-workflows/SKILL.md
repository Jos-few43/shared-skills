# GitOps Workflows — SOPS+age Secrets Management

Implements GitOps patterns with encrypted secrets that integrate with the existing `~/secrets/` infrastructure. Manages `.sops.yaml` configuration, encrypted file lifecycle, and deployment sync verification.

## Usage

```
/gitops-workflows init
/gitops-workflows encrypt <file>
/gitops-workflows decrypt <file>
/gitops-workflows diff <commit1> <commit2>
/gitops-workflows sync
/gitops-workflows sync --fix
```

## Existing Secrets Infrastructure

This skill integrates with the existing setup:

| Component | Path | Purpose |
|---|---|---|
| Age key | `~/secrets/keys/` | Primary encryption key |
| SOPS store | `~/secrets/store.yaml` | SOPS-encrypted secret values |
| Manifests | `~/secrets/manifests/` | Per-container secret definitions |
| CLI | `~/bin/secrets` | Secret management (set, inject, rotate, check) |

Always use `secrets set` + `secrets inject` for managing secret values. This skill manages the GitOps infrastructure layer: `.sops.yaml` config, file encryption state, and git integration.

## Subcommand: `init`

Set up SOPS+age configuration for a repository.

### Steps

1. Locate the age public key:

```bash
# Find age public key from existing keys directory
ls ~/secrets/keys/
# Typical filename: age.agekey or similar
# Extract the public key (starts with "age1...")
grep "^# public key:" ~/secrets/keys/*.agekey 2>/dev/null \
  || grep "^age1" ~/secrets/keys/*.pub 2>/dev/null \
  || age-keygen -y ~/secrets/keys/*.agekey 2>/dev/null
```

2. Determine the repository root (current directory or argument).

3. Write `.sops.yaml` at the repo root:

```yaml
# .sops.yaml — SOPS encryption configuration
# Generated by /gitops-workflows init
creation_rules:
  # Secrets and credentials — encrypt all sensitive patterns
  - path_regex: .*\.env$
    age: <age-public-key>

  - path_regex: .*secrets.*\.ya?ml$
    age: <age-public-key>

  - path_regex: .*credentials.*\.ya?ml$
    age: <age-public-key>

  - path_regex: .*\.secret$
    age: <age-public-key>

  # Config overlays with sensitive values
  - path_regex: overlays/.*/secrets\.ya?ml$
    age: <age-public-key>

  - path_regex: environments/.*/secrets\.ya?ml$
    age: <age-public-key>
```

4. Write `.gitattributes` diff driver entry (for readable diffs):

```
# .gitattributes
*.env diff=sopsdiffer
*secrets*.yaml diff=sopsdiffer
*secrets*.yml diff=sopsdiffer
```

5. Register the sops diff driver (run once per machine):

```bash
git config diff.sopsdiffer.textconv "sops --decrypt"
```

6. Write a pre-commit hook to block unencrypted secrets:

```bash
cat > .git/hooks/pre-commit << 'HOOK'
#!/bin/bash
# Pre-commit: block unencrypted files matching .sops.yaml patterns

SOPS_CONFIG=".sops.yaml"
if [ ! -f "$SOPS_CONFIG" ]; then
  exit 0
fi

FAILED=0

# Check staged files against sops creation_rules patterns
for file in $(git diff --cached --name-only); do
  [ -f "$file" ] || continue

  # Check if file matches a sops pattern
  if grep -qP "path_regex:" "$SOPS_CONFIG"; then
    patterns=$(grep "path_regex:" "$SOPS_CONFIG" | sed 's/.*path_regex: //')
    for pattern in $patterns; do
      if echo "$file" | grep -qE "$pattern"; then
        # File matches a sops pattern — verify it's encrypted
        if ! sops --decrypt "$file" > /dev/null 2>&1; then
          # Not encrypted — check if it contains sensitive data
          if grep -qiE "(password|secret|api.?key|token|private.?key|credential)" "$file" 2>/dev/null; then
            echo "ERROR: $file matches sops pattern and appears to contain sensitive data but is not encrypted."
            echo "       Run: /gitops-workflows encrypt $file"
            FAILED=1
          fi
        fi
      fi
    done
  fi
done

exit $FAILED
HOOK
chmod +x .git/hooks/pre-commit
```

7. Report: print the `.sops.yaml` written, confirm hook installed, show the age public key fingerprint used.

## Subcommand: `encrypt`

Encrypt a plaintext file with sops, respecting `.sops.yaml` creation rules.

### Steps

1. Verify `.sops.yaml` exists in the repo root. If not, suggest running `init` first.
2. Verify the file is not already encrypted:

```bash
sops --decrypt <file> > /dev/null 2>&1 && echo "already encrypted" || echo "plaintext"
```

3. Confirm the file matches a `.sops.yaml` pattern. Warn if it does not (file will encrypt but with no config-driven key selection).

4. Encrypt in-place:

```bash
sops --encrypt --in-place <file>
```

5. Verify encryption succeeded:

```bash
# Should fail (can't read plaintext) if encrypted correctly
grep -qiE "(password|secret|api.?key|token)" <file> && echo "WARNING: plaintext values still visible" || echo "Encryption verified"
```

6. Stage the encrypted file:

```bash
git add <file>
```

7. Report: confirm encryption, show the sops metadata header (creation date, age recipient), remind user to commit.

### Safety Rules

- Never print decrypted values to the terminal.
- If the file already contains `sops:` metadata, do not re-encrypt — it is already encrypted.
- If encryption fails, check that the age key in `.sops.yaml` matches `~/secrets/keys/`.

## Subcommand: `decrypt`

Decrypt an encrypted file for viewing or editing. Does not write plaintext to disk permanently.

### Steps

1. Verify the file is sops-encrypted:

```bash
sops --decrypt <file> > /dev/null 2>&1 || { echo "File is not sops-encrypted"; exit 1; }
```

2. For viewing only (pipe to pager, do not write):

```bash
sops --decrypt <file> | less
```

3. For editing (sops opens in $EDITOR and re-encrypts on save):

```bash
sops <file>
```

4. Never write decrypted output to a `.env` or plaintext file — use `secrets set` + `secrets inject` to update the secrets store.

### Safe Editing Pattern

```bash
# Correct: sops manages encrypt/decrypt/re-encrypt cycle
sops ~/secrets/store.yaml

# Correct: use the secrets CLI for individual values
secrets set MY_KEY "new-value"
secrets inject <container>

# Wrong: never do this
sops --decrypt store.yaml > store-plaintext.yaml   # plaintext on disk!
```

## Subcommand: `diff`

Show a human-readable diff of encrypted files between two git refs.

### Steps

1. Parse arguments: `diff <ref1> <ref2>` or `diff <ref>` (compares ref to HEAD).

2. For each encrypted file changed between refs, decrypt both versions and diff:

```bash
# List changed encrypted files between two refs
git diff --name-only <ref1> <ref2> | while read file; do
  # Check if file is sops-encrypted in either version
  git show <ref1>:"$file" 2>/dev/null | grep -q "sops:" || continue

  echo "=== $file ==="
  diff \
    <(git show <ref1>:"$file" | sops --decrypt /dev/stdin 2>/dev/null) \
    <(git show <ref2>:"$file" | sops --decrypt /dev/stdin 2>/dev/null) \
    || true
  echo ""
done
```

3. For a single commit diff against its parent:

```bash
git diff --name-only <ref>~1 <ref> | while read file; do
  git show <ref>:"$file" 2>/dev/null | grep -q "sops:" || continue
  echo "=== $file (changed in <ref>) ==="
  diff \
    <(git show <ref>~1:"$file" 2>/dev/null | sops --decrypt /dev/stdin 2>/dev/null) \
    <(git show <ref>:"$file" | sops --decrypt /dev/stdin 2>/dev/null) \
    || true
done
```

4. Redact high-entropy values from diff output before displaying (show key names, not values where possible).

## Subcommand: `sync`

Verify all files matching `.sops.yaml` patterns in the working tree are actually encrypted. Identify any plaintext files that should be encrypted.

### Steps

1. Read `.sops.yaml` and extract all `path_regex` patterns.

2. Find all tracked and untracked files matching each pattern:

```bash
# Tracked files
git ls-files | grep -E "<pattern>"

# Untracked files (potential leaks)
git ls-files --others --exclude-standard | grep -E "<pattern>"
```

3. For each matched file, check encryption state:

```bash
sops --decrypt <file> > /dev/null 2>&1 && echo "ENCRYPTED: $file" || echo "PLAINTEXT: $file"
```

4. Check `.gitignore` — warn if a plaintext secrets file is only protected by `.gitignore` (not encrypted):

```bash
git check-ignore <file> && echo "GITIGNORED (not encrypted): $file — consider encrypting"
```

5. Report findings in a table:

```
Sync Results for <repo>
=======================

ENCRYPTED (safe to commit):
  ✓ secrets/store.yaml
  ✓ overlays/prod/secrets.yaml

PLAINTEXT — MUST ENCRYPT before commit:
  ✗ .env                  → run: /gitops-workflows encrypt .env
  ✗ config/credentials.yaml → run: /gitops-workflows encrypt config/credentials.yaml

GITIGNORED (plaintext, protected only by .gitignore):
  ~ .env.local             → consider encrypting or ensuring .gitignore is committed

UNTRACKED sensitive files (not in git):
  ~ deploy/secrets.yaml    → not tracked; verify this is intentional
```

6. With `--fix`: automatically encrypt all PLAINTEXT files listed above:

```bash
for file in <plaintext-files>; do
  sops --encrypt --in-place "$file" && git add "$file"
  echo "Encrypted and staged: $file"
done
```

## GitOps Config Repo Structure

When initializing a new config repo, use this layout:

```
config-repo/
├── .sops.yaml               # Encryption rules (committed, no secrets)
├── .gitattributes           # Diff driver registration
├── base/                    # Base Kustomize or Helm configs
│   ├── deployment.yaml
│   └── service.yaml
├── overlays/                # Per-environment overrides
│   ├── dev/
│   │   └── kustomization.yaml
│   ├── staging/
│   │   ├── kustomization.yaml
│   │   └── secrets.yaml     # ← SOPS-encrypted
│   └── prod/
│       ├── kustomization.yaml
│       └── secrets.yaml     # ← SOPS-encrypted
└── environments/            # Alternative: flat environment structure
    ├── dev.yaml
    ├── staging.yaml         # ← SOPS-encrypted if contains secrets
    └── prod.yaml            # ← SOPS-encrypted if contains secrets
```

## Safety Rules

1. Never commit plaintext secrets — the pre-commit hook enforces this, but always verify with `sync` before pushing.
2. Always verify encryption before `git add`: `sops --decrypt <file> > /dev/null 2>&1 && echo ok`.
3. Never decrypt to a file — use `sops <file>` for in-place editing, or pipe to pager for reading.
4. If `.sops.yaml` doesn't cover a file pattern that contains sensitive data, update it and re-run `init`.
5. The age key at `~/secrets/keys/` is the single source of truth — back it up securely and never commit it.
6. Rotate keys via `secrets rotate` — do not manually edit the age key.
7. After adding a new team member's age key to `.sops.yaml`, re-encrypt all secrets: `sops updatekeys <file>`.

## Integration with Secrets CLI

| Task | Command |
|---|---|
| Add/update a secret value | `secrets set MY_KEY "value"` |
| Inject secrets into containers | `secrets inject <container>` |
| List all secret keys | `secrets list` |
| Check for expiring secrets | `secrets check` |
| Find unmanaged plaintext .env files | `secrets lint --scan` |
| Rotate a secret | `secrets rotate MY_KEY` |

The `~/bin/secrets` CLI manages values inside `~/secrets/store.yaml`. This skill manages the GitOps infrastructure (`.sops.yaml`, file encryption state, git integration) that wraps around it.
