# Secrets Management

## Overview

All credentials are managed by a SOPS+age encrypted store at `~/secrets/store.yaml` with a CLI at `~/bin/secrets`. Never edit `.env` files directly — use the CLI to set secrets and inject them into container manifests.

## CLI Reference

```bash
secrets list                        # Show all secrets (keys only, grouped by namespace)
secrets get <key>                   # Decrypt and print a single value
secrets set <key> <value>           # Add/update a secret
secrets set <key> <val> --expires YYYY-MM-DD  # With expiry tracking
secrets remove <key>                # Delete a secret
secrets inject <container>          # Generate .env from manifest
secrets inject --all                # Inject all containers
secrets inject --dry-run <container> # Preview without writing
secrets rotate <key>                # Interactive rotation (prompts for new value)
secrets check                       # Report secrets expiring within 30 days
secrets audit [--tail N]            # Show recent operations
secrets lint [--scan]               # Verify integrity + find unmanaged .env files
```

## Key Naming Convention

Secrets use `namespace.key_name` dotted format:

| Namespace | Service |
|-----------|---------|
| `litellm_blue` | Blue LiteLLM instance (port 4001) |
| `litellm_green` | Green LiteLLM instance (port 4002) |
| `litellm_ui` | LiteLLM UI credentials |
| `openclaw` | OpenClaw infrastructure |
| `open_notebook` | Open Notebook app |
| `finops` | FinOps/Grafana stack |

## Adding a New Service

When adding a new container or service that needs credentials:

1. **Add secrets to the store:**
   ```bash
   secrets set <namespace>.<key> "<value>"
   ```

2. **Create a manifest** at `~/secrets/manifests/<service>.yaml`:
   ```yaml
   container: <service-name>
   target: ${HOME}/path/to/.env
   format: env
   secrets:
     ENV_VAR_NAME: namespace.key_name
   ```

3. **Inject:**
   ```bash
   secrets inject <service-name>
   ```

4. **Verify:**
   ```bash
   secrets lint
   ```

## Rotating a Key

```bash
# 1. Rotate in store (interactive prompt)
secrets rotate api_keys.groq

# 2. CLI shows affected containers — re-inject them
secrets inject litellm-blue
secrets inject litellm-green

# 3. Restart affected services
```

## Paths

| Path | Purpose |
|------|---------|
| `~/secrets/store.yaml` | Encrypted store (SOPS+age, 600 perms) |
| `~/secrets/manifests/` | Container manifests (plaintext YAML) |
| `~/secrets/.sops.yaml` | SOPS config (age public key) |
| `~/secrets/audit/audit.log` | JSON audit trail (gitignored) |
| `~/bin/secrets` | CLI entry point |
| `~/.config/age/key.txt` | Age private key (600 perms) |

## Rules

- **Never** edit `.env` files directly — they are generated by `secrets inject`
- **Never** commit `store.yaml` decrypted — SOPS encrypts values automatically
- **Always** use manifests for new services
- **Always** set `--expires` on API keys when known
- **Run** `secrets lint --scan` after adding new projects to catch unmanaged secrets
- Store file and all generated `.env` files must have 600 permissions

## Troubleshooting

**"sops failed":** Check that `~/.config/age/key.txt` exists and has 600 perms. Verify `~/bin/sops --version` returns 3.9.4.

**"Manifest not found":** The manifest file must be at `~/secrets/manifests/<container-name>.yaml`. Name must match exactly.

**Lint false positives:** Template `.env` files with placeholder values (like `REPLACE_WITH_*`) may trigger warnings. These are safe to ignore.

**Permissions reset:** If a `.env` file gets wrong permissions, re-run `secrets inject <container>` — it always sets 600.
